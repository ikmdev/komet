= View Coordinate Architecture Migration Plan
:toc: macro
:toclevels: 3
:icons: font

Tracking document for migrating from the legacy `KometNodeFactory` / `ViewProperties`
pattern to the `KlView.Factory` / `KlContext` architecture in the knowledge-layout
subproject. This refactoring spans `framework`, `kview`, and `knowledge-layout`.

toc::[]

== Background: Two Architectures, One Transition

=== The intended design

The application needs a hierarchy of view coordinates:

----
Knowledge Base  (root defaults)
  └─ Session    (user preferences)
      └─ Journal  (journal-level overrides, e.g. time-travel)
          └─ Window   (window-level overrides)
              └─ Pane  (pane-level overrides, e.g. historical vs current)
----

Each level can override STAMP coordinates, language, logic, navigation, or edit
coordinates. Fields not overridden are inherited from the parent. This enables
capabilities like time-travel (one window historical, another current), resolving
contradictions between authors or branches, and per-pane coordinate customization.

=== The knowledge-layout system already solves this

The `KlContext` + `KlView` architecture handles the multi-level override chain
*without `ViewProperties` at all*. The mechanism is fundamentally different and cleaner:

`KlView` resolves coordinates by walking the JavaFX node tree:

----
KlView.context(node)
  → check node.properties[KL_CONTEXT]
  → walk up parent chain
  → check scene
  → check window
  → fallback: KnowledgeBaseContext
----

Each level (Window -> Scene -> Area -> child Area) can have its own `KlContext` with
its own `ObservableView viewCoordinate()`. A component calls `viewForContext()` or
`calculatorForContext()` and gets the nearest context's coordinate -- naturally
inheriting from ancestors if no local context is set.

This is *exactly* the intended hierarchy (knowledge base -> session -> journal ->
window -> pane) but implemented through the JavaFX scene graph rather than through a
hand-wired `ViewProperties` parent chain. No casts. No `ObservableViewNoOverride` vs
`ObservableViewWithOverride` distinction needed at the API boundary.

=== What `KometNodeFactory` gets wrong

[cols="1,2,2", options="header"]
|===
| Concern
| KometNodeFactory (old)
| KlView.Factory (new)

| View coordinate source
| Passed as `ObservableViewNoOverride` parameter -- assumes flat 2-level hierarchy
| Resolved from JavaFX tree via `KlContext` -- naturally N-level

| Factory parameter
| Requires the root view + creates one override
| No view parameter -- component discovers its context after insertion into scene graph

| Restoration
| `@Reconstructor` + `Encodable.decode` reflection
| `KlRestorable` + preference-based `FACTORY_CLASS_NAME`

| Lifecycle
| None -- constructor does everything
| `knowledgeLayoutBind()` / `knowledgeLayoutUnbind()` separate setup from construction

| Discovery
| `ServiceLoader<KometNodeFactory>`
| `PluggableService.forName()` from preferences
|===

The key insight: *`KometNodeFactory` forces the view coordinate to be known at
construction time and passed as a parameter*, which requires the caller to manually
assemble the override chain. `KlView` defers coordinate resolution to the scene graph,
which means the chain assembles itself.

=== The collision point (ClassCastException)

`ConceptKlWindow` sits at the boundary between architectures. It is called from the new
pattern (`KlWindowPreferencesUtils.getJournalViewProperties()` builds a multi-level
chain producing `ViewProperties` whose parent is an `ObservableViewWithOverride`) but
internally delegates to the old pattern (`KometNodeFactory.create()` which demands
`ObservableViewNoOverride`). The cast fails at runtime.

This is a symptom of the broader problem: the `ViewProperties` class conflates the
*root view* (needed by old code) with the *immediate parent in the override chain*
(needed by new code). With multi-level overrides, these are no longer the same object.

== Layout Area Architecture: Applying the MGC Pattern

=== Design decision: `KlAreaForBoolean` over `KlBooleanField`

The codebase contains two parallel interface hierarchies for field areas under `KlArea`:
`KlAreaForFeature` (with subtypes like `KlAreaForBoolean<FX>`) and `KlField` (with
subtypes like `KlBooleanField`). After analysis, we consolidate toward **`KlAreaForBoolean`**
(dropping its `<FX>` generic) rather than `KlBooleanField`, for these reasons:

1. **Naming consistency**: `KlAreaForBoolean` fits the `KlArea` naming family.
   `KlBooleanField` is an orphan name that doesn't follow the established convention.

2. **Infrastructure**: The `KlAreaForBoolean` side has full Blueprint implementations
   (`EditableBooleanFieldArea`, `ReadOnlyBooleanFieldArea`) with lifecycle management,
   preferences save/restore, context subscription, and ServiceLoader registration.
   The `KlBooleanField` side has `BaseDefaultKlField` implementations with none of this.

3. **Scene graph integration**: The `KlArea` hierarchy provides `fxObject()`, `context()`,
   `properties()`, `calculatorForContext()` -- the full scene-graph-based coordinate
   resolution system. This must be available at the marker/API layer.

=== The `<FX>` type parameter: moving it to the right layer

`KlAreaForBoolean<FX extends Region>` currently carries an `<FX>` type parameter that
tracks the specific JavaFX control type. Analysis shows this parameter is:

- **Never used externally**: `FieldPropertiesPane` (the only external consumer) erases
  it via `ServiceLoader.load(KlAreaForBoolean.Factory.class)` -- `.class` literals and
  wildcard iteration erase all generics.
- **Only used internally**: Every `getFxPeer()` call is inside the concrete class itself
  (e.g., `EditableBooleanFieldArea` calling `getFxPeer().valueProperty()`). The class
  already knows its control type because it created it in the constructor.
- **Valuable in the abstract layer**: `EditableFieldAreaBlueprint<DT, FX>` uses `<FX>` to
  provide type-safe `getFxPeer()` without casting. This is the right place for it.

Following the **Marker-Generic-Concrete (MGC) pattern**
(see `framework/.../observable/adoc/MarkerGenericConcrete.adoc`):

----
Layer 1 (Marker):    KlAreaForBoolean             ← generic-free, consumer-facing
Layer 2 (Generic):   EditableFieldAreaBlueprint<DT, FX>  ← tracks FX internally
Layer 3 (Concrete):  EditableBooleanFieldArea      ← final, no generics in declaration
----

This means `KlArea` itself drops `<FX>`:

----
Current:  KlArea<FX extends Region> extends KlView<FX>, KlPeerToRegion<FX>
Target:   KlArea extends KlView<Region>, KlPeerToRegion<Region>
----

`KlView<FX>` keeps `<FX>` because it legitimately handles Window, Scene, and Region.
`KlArea` fixes it to `Region` at the marker level. Concrete classes get the specific
control type through covariant returns from their abstract superclass.

=== Resolving the `KlPeerToRegion` conflict

Dropping `<FX>` from `KlArea` creates a potential type conflict: a concrete class would
inherit both `KlPeerToRegion<Region>` (from `KlArea`) and `KlPeerToRegion<KLBooleanControl>`
(from `AreaBlueprint<KLBooleanControl>`). Java prohibits implementing the same generic
interface with different type arguments.

The fix: `AreaBlueprint<FX>` implements `KlPeerToRegion<Region>` (not `<FX>`):

[source,java]
----
// Layer 1: KlArea (no FX)
public sealed interface KlArea
    extends KlView<Region>, KlPeerToRegion<Region> { ... }

// Layer 2: AreaBlueprint (typed internally, Region externally)
public abstract class AreaBlueprint<FX extends Region>
    extends StateAndContextBlueprint<FX>
    implements KlPeerToRegion<Region> {  // ← Region, not FX

    // Internal typed accessor for subclasses
    protected final FX getFxPeer() { return (FX) fxObject(); }
}
----

`StateAndContextBlueprint<KLBooleanControl>.fxObject()` returns `KLBooleanControl`,
which satisfies `Region fxObject()` from the interface via covariant return types.
Java generates a bridge method automatically.

== Phase 7: MGC Field Area Consolidation

This phase consolidates the duplicate field implementations under the Blueprint
hierarchy and applies the MGC pattern by removing `<FX>` from the interface layer.
The work is ordered to minimize wasted effort: consolidate first (delete duplicates
while interfaces are unchanged), then drop `<FX>` (touching only the surviving files).

=== Phase 7a: Consolidate duplicate implementations (kview only)

Blueprint-based implementations. No interface changes -- only kview is touched.

* `KlRestorable` interface for persistence
* kview controls (`Toast`, `ToggleSwitch`, events)
* Framework utilities (`Identicon`, `TimeUtils`, `DragImageMaker`) -- low coupling, can stay

=== Impact of Phase 7 on the layout editor

**Phase 7a (consolidation)** has **no impact** on the layout editor. It only deletes
kview duplicates and redirects `KlFieldHelper` -- the layout editor does not use
`KlFieldHelper` or `BaseDefaultKlField`.

**Phase 7b (drop `<FX>`)** has a **small positive impact** on the layout editor:

* `FieldPropertiesPane` currently calls `ServiceLoader.load(KlAreaForBoolean.Factory.class)`.
  After Phase 7b, `KlAreaForBoolean.Factory` drops its `<FX>` type parameter but the
  `.class` reference is unchanged. The code continues to work with no modifications needed.
* The simplified factory types (no generics) make future enhancements easier.

**Phase 7c (deprecate `KlField`)** has **no impact** on the layout editor. It does not
reference any `KlField` types.
